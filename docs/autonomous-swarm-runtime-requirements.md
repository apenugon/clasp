# Autonomous Swarm Runtime Requirements

This document captures the concrete Clasp features needed to run a native, self-hosted swarm that can improve Clasp itself and compete on AppBench-class workloads.

## Goal

Run a swarm entirely through native `claspc` and the native runtime, with:

- no Haskell in the execution loop
- no Node/Bun/JS server-side helpers
- one native runtime
- one native `claspc`
- one evented control plane that can plan, code, verify, review, benchmark, and merge bounded work

## System Shape

The intended architecture is:

- `manager`
  - owns task DAGs, leases, budgets, priorities, approvals, and stop conditions
- `planner`
  - decomposes goals into bounded tasks with explicit inputs and success criteria
- `coder`
  - edits code in isolated workspaces
- `reviewer`
  - checks correctness, regressions, and design alignment
- `verifier`
  - runs focused checks and scenario-level verification
- `optimizer`
  - generates competing variants and benchmark-oriented improvements
- `integrator`
  - selects winners, merges, requeues failures, and records outcomes
- `memory`
  - stores prior failures, benchmark deltas, design constraints, and reusable artifacts

External users should talk only to the `manager`, via:

- `claspc swarm ...`
- a native HTTP control plane
- an optional browser UI backed by that control plane

## Required Native Runtime Features

### 1. Event Log

The swarm needs a native append-only event log for:

- task creation
- lease acquisition/release
- worker heartbeat
- patch publication
- verification pass/fail
- benchmark deltas
- merge decisions
- approvals and escalations

Requirements:

- durable on local disk first
- ordered and replayable
- stable IDs for tasks, runs, agents, and artifacts

### 2. State Store

The runtime needs a native structured state store derived from the event log:

- task DAG
- agent leases
- retry counters
- active budgets
- artifact references
- memory entries
- approvals
- benchmark score history

SQLite is the practical first implementation, and the repo now has that as the first-class `claspc swarm` store.

### 3. Artifact Store

Large outputs should not be passed around as prompt text or retained without bounds.

The runtime needs native artifact storage for:

- diffs
- logs
- benchmark traces
- compiled outputs
- screenshots
- review reports

Artifacts should be addressable by ID and referenced from task state.

### 4. Lease / Heartbeat Model

Agents must not own work forever.

The runtime needs:

- task leases
- lease expiry
- worker heartbeat
- dead-worker reaping
- retry scheduling
- bounded retry policy

### 5. Native Workflow / Supervisor Runtime

The swarm is fundamentally a workflow system.

The native runtime needs:

- durable workflow execution
- supervisor trees
- restart policies
- mailbox / event dispatch
- bounded child spawning
- safe stop / resume
- state handoff and upgrade-safe transitions

### 6. Native Tool Execution

Agents need controlled execution of tools and compilers.

The runtime needs:

- process spawning
- bounded stdout/stderr artifact capture
- timeouts
- environment control
- working-directory control
- exit-status capture
- policy-gated filesystem/network/process permissions

The repo now has the first native slice of this through `claspc swarm tool ...` and `claspc swarm verifier run ...`, with durable run records and artifact capture.

### 7. Native Merge / Review Gates

The runtime needs first-class support for:

- verifier surfaces
- merge gates
- approval gates
- policy-backed escalation
- merge decision recording

The repo now has the first native mergegate slice through `claspc swarm mergegate decide ...`, backed by stored verifier runs.

### 8. Native Route / Control-Plane Runtime

The swarm needs a server-side control plane implemented natively, including:

- run creation
- run status
- event streaming
- approval endpoints
- stop / resume / reprioritize
- artifact retrieval

The repo now has the first manager-facing native slice here through:

- `claspc swarm start|status|history|tasks|summary|tail`
- `claspc swarm stop|resume`
- `claspc swarm runs|artifacts`
- `claspc swarm approve|approvals`
- `claspc swarm policy set`
- `claspc swarm manager next`
- `claspc swarm objective create|status`
- `claspc swarm objectives`
- `claspc swarm task create`
- `claspc swarm ready`
- `claspc swarm memory put|query|search`

That native slice now also includes:

- objective-scoped task count budgets
- task-scoped and objective-scoped run budgets
- persisted dependency edges for task DAGs
- task and objective deadlines
- task lease-timeout metadata
- dependency-aware ready-set projection
- lease-expiry-aware lease reacquisition checks
- persisted merge-policy requirements with approval/verifier state projection
- objective-driven manager next-action projection
- ordinary Clasp context-pack assembly over task state, manager projection, verifier traces, artifacts, and ranked memory

## Required Compiler / Language Features

### 1. Whole-Project Native Build Ownership

`claspc` should decide frontend vs backend outputs automatically and build the whole project.

Needed compiler features:

- explicit whole-project build planning
- frontend/backend surface classification
- packaging of native backend binaries
- packaging of client-side JS assets only where frontend code exists

### 2. Fast Iterative Native Rebuilds

Swarm development requires fast rebuilds.

Needed compiler features:

- parallel project bundle construction
- parallel native image section work
- per-module native image caching
- persistent incremental compilation keyed by module dependency/interface fingerprints
- deterministic output after parallel work

### 3. Strong Native Surface Coverage

The self-hosted/native compiler must own these surfaces:

- routes
- workflows
- hooks
- tools
- tool servers
- guides
- policies
- agents
- verifiers
- merge gates

No backend semantics should depend on JS helper files.

The ordinary Clasp swarm library now has reusable evidence-pack and context-pack surfaces that let an agent assemble compact task evidence from native swarm state without scraping logs by hand. `taskEvidencePack` exposes the reusable retrieval subset for standalone agent backends: local ranked memory, objective-level benchmark history, dependency task ids/statuses/ready flags, query-ranked artifact search matches, dedicated semantic-index artifact matches for compiler-owned `semanticIndex.entries`, verifier run traces, artifact references, bounded artifact excerpts, and mailbox summaries. Semantic index artifacts are now a typed ordinary-Clasp surface (`SwarmSemanticIndexEntry` plus `artifactWriteSemanticIndex`) instead of hand-written JSON strings, so Clasp-native builders/planners can publish searchable edit-file, artifact-reference, and surface-id hints directly into the artifact store. Compiler-owned `claspc semantic` summaries can also be converted directly into typed swarm semantic-index artifacts through `artifactWriteSemanticSummaryIndex`, which lets ordinary Clasp agents ingest source modules, definitions, routes, schemas, host boundaries, and source files without bespoke JSON scraping. Evidence and context packs now project dependencies and semantic entries into direct `dependencyTaskIds`, `dependencyTaskStatuses`, `dependencyTaskReady`, `semanticIndexEntryIds`, `semanticIndexEditFiles`, `semanticIndexArtifactRefs`, `semanticIndexSurfaceIds`, and `semanticIndexQueryTexts` fields, so local agent backends can route work from typed evidence instead of scraping JSON excerpts. Context-pack assembly re-reads semantic-index artifacts through a separate bounded full-artifact path before decoding, so larger compiler summary indexes do not depend on the smaller prompt-excerpt limit. `taskContextPack` layers ready-set projection and manager next-action projection on top of the same evidence pack. The native FeedbackLoop and GoalManager planner now render dependency ids/statuses/ready flags, semantic projections, memory, artifacts, run traces, and benchmark evidence directly into builder, verifier, and planner prompt artifacts, and persist verifier feedback plus GoalManager benchmark checkpoints into swarm memory, so standalone Clasp agent backends receive the same durable state projection as Codex-backed agents.

### 4. Better Compiler-Authoring Ergonomics

The swarm will spend a lot of time writing compiler/runtime code in Clasp.

High-priority language improvements:

- multiline list and record literals
- trailing commas everywhere
- record destructuring / update
- collection helpers:
  - `map`
  - `fold`
  - `filter`
  - `find`
  - `any`
  - `all`
  - `concat`
- typed integer helpers for retry counters and bounded loops:
  - `intAdd`
  - `intSubtract`
- dictionary / map type
- clearer unsupported-surface diagnostics

### 5. Native Memory and Task APIs

The language/runtime surface should expose native operations for:

- event append/read
- task lookup/update
- artifact read/write
- lease acquire/renew/release
- benchmark result recording
- child worker spawn/reap

These should be Clasp-level primitives over the native runtime, not external shell glue.

The repo now has an ordinary-program slice of this through the internal `@swarm` runtime lane, which is wrapped by ordinary Clasp code in [`examples/swarm-native/Swarm.clasp`](/home/akul_medexfinance_com/clasp/examples/swarm-native/Swarm.clasp). That path lets Clasp programs create objectives/tasks with run budgets and deadlines, acquire and release leases, reprioritize queued tasks, inspect priority-ordered ready/state projections, run tools/verifiers, inspect runs/artifacts, publish bounded task artifacts, read and search artifact content through bounded native retrieval, store/query/search durable swarm memory, record benchmark results into objective memory, and drive approvals and merge decisions without shell wrapper scripts or external `claspc swarm ...` subprocesses. Native manager actions now expose the selected task priority, and ready/objective/task projections sort higher-priority ready work before older lower-priority work, so ordinary Clasp planners can change scheduling policy at runtime without changing the compiler. For the current supervised single-node swarm model, task creation plus durable tool/verifier execution is the native child-work substrate. Native ranked memory and artifact search cover local evidence retrieval, and `Swarm.clasp` now adds typed sparse semantic-memory embeddings (`SwarmSparseEmbedding`, `SwarmEmbeddedMemoryValue`, and `swarmEmbeddedMemorySearchFromMemory`) so a standalone agent can persist provider- or local-model embedding dimensions in durable memory values, decode them safely, and rank memories by model-scoped semantic overlap without falling back to raw text search. A provider-style embedding trust boundary (`SwarmEmbeddingProviderRequest`, `UntrustedEmbeddingProviderOutput`, and `validateWeightedEmbeddingProviderOutput`) now keeps embedding outputs untrusted until provider/model/status/schema fields and fixed-point payloads validate, and `SwarmWeightedVectorIndex` plus `swarmWeightedVectorIndexFromMemory`, `swarmWeightedVectorIndexText`, `swarmWeightedVectorIndexFromText`, and `swarmWeightedVectorIndexSearch` give ordinary programs a JSON-persistent weighted vector index for deterministic local or future remote embedding outputs. `SwarmWeightedVectorStore`, `swarmWeightedVectorStoreFromIndexes`, `swarmWeightedVectorStoreText`, `swarmWeightedVectorStoreFromText`, `swarmWeightedVectorStoreShardCountForQuery`, and `swarmWeightedVectorStoreSearch` add a JSON-persistent sharded store that keeps provider/model/scale vector indexes separated while searching only the matching shard. `SwarmEmbeddingProviderCommandTransport`, `swarmEmbeddingProviderCommand`, and `swarmRunEmbeddingProviderCommand` add a command-template transport that renders provider/model/input/scale placeholders, invokes an external embedding command, and validates its output through the same trust boundary. `SwarmEmbeddingProviderNetworkAdapter`, `SwarmEmbeddingProviderNetworkAdapterSummary`, `swarmEmbeddingProviderNetworkAdapterValidationMessage`, `swarmEmbeddingProviderNetworkAdapterSummary`, `swarmEmbeddingProviderNetworkAdapterCommandTransport`, `swarmEmbeddingProviderBearerNetworkAdapter`, `swarmOpenAiEmbeddingProviderNetworkAdapter`, `swarmEmbeddingProviderNetworkAdapterCommand`, and `swarmRunEmbeddingProviderNetworkAdapter` add first-class authenticated/network adapter configuration and reusable provider profiles for endpoint, auth env, allowlisted destination, command rendering, request provider/model matching, common bearer/OpenAI-compatible setup, timeout, and untrusted-output validation. Native/provider-specific HTTP adapter execution and large-scale approximate/vector-database persistence remain follow-on work rather than blockers for the ordinary-program orchestration loop. Typed integer helpers now back ordinary Clasp retry counters, so managed loops no longer depend on hand-written finite counter tables.

The feedback-loop fixture also now proves a non-Codex backend path with [`examples/swarm-native/LocalAgent.clasp`](/home/akul_medexfinance_com/clasp/examples/swarm-native/LocalAgent.clasp): the native builder/verifier loop can invoke a Clasp program as the agent backend through the generic agent-command template, persist durable prompt files and reports, retry from verifier feedback, and converge without a Codex binary. That gives the Clasp-native planner/child-agent path task-specific standalone behavior instead of only generic plumbing proof.

Agent backend command construction is now factored into an ordinary Clasp API in [`examples/swarm-native/AgentBackend.clasp`](/home/akul_medexfinance_com/clasp/examples/swarm-native/AgentBackend.clasp). It also exposes typed `AgentBackendPolicy` values, including `agentBackendStandalonePolicy`, plus `AgentBackendPolicySummary` evidence records that show the active policy name, backend kind, prompt transport, validity, primary message, all validation messages, exact transport requirements, blocking gaps, required closure steps, missing placeholders, and the reusable `agentBackendStandaloneRecommendedTemplate` command shape. The same module now exposes `agentBackendClaspRunTemplate`, `agentBackendLocalAgentTemplate`, and `agentBackendLocalPlannerTemplate`, which render the standard `run <program> -- --role ... --schema ... --report ... --prompt-path ... --workspace ...` shape for ordinary Clasp programs. That gives a standalone swarm a reusable no-shell-wrapper template for making a Clasp program the planner, builder, or verifier backend while still satisfying the same backend policy checks.

Provider/model output now has a first explicit ordinary-Clasp trust boundary in [`examples/swarm-native/ModelBoundary.clasp`](/home/akul_medexfinance_com/clasp/examples/swarm-native/ModelBoundary.clasp). `ModelBoundaryRequest` keeps the prompt as a typed `Prompt`, `UntrustedModelOutput` records raw provider text with `trusted = false`, and `validatePlannerModelOutput` is the only fixture path that turns raw planner output into `TrustedPlannerModelOutput`. That validator checks provider/model/schema/status, then uses `tryDecode PlannerReport`, a required-field guard, and planner-shape checks before marking the report as `trust = "validated"`. The GoalManager planner-report readers now mirror that boundary in the real monolithic and modular planner paths with `TrustedPlannerReport` and `PlannerReportBoundarySummary`, so current and legacy planner reports are decoded through `tryDecode` before they become executable plans, while malformed or schema-invalid model text stays summarized as `trust = "untrusted"`. The JS and native harnesses reject missing fields, malformed output, schema mismatches, failed provider statuses, and any raw output that arrives pre-marked trusted, so future planner/provider integrations have an executable pattern that keeps model text untrusted until schema validation succeeds.

Planner resume/reuse state follows the same recovery rule: current and legacy `planner-input.json` state, child-loop status files, watched-process heartbeat JSON, mailbox state, benchmark checkpoints, benchmark signal output, and ordinary agent-handoff snapshots are read through `tryDecode` or a safe fallback path, so corrupted recovery files become ordinary manager errors or empty projections instead of unchecked decode crashes during a standalone swarm resume.

The native feedback-loop prompt contract is now prompt-contained for the task itself: builder and verifier prompts include the task file text as well as the durable task path and native context pack. That lets prompt-only or Clasp-native agent backends understand the assigned task without relying on a separate task-file read.

## Required Safety / Governance Features

The swarm should be autonomous but bounded.

Required controls:

- explicit task budgets
- maximum spawn depth
- capability-based tool access
- policy-backed filesystem/network/process permissions
- approval-required actions for risky merges or broad writes
- audit trail for every merge decision

## Current Gaps

At the time of writing, Clasp is moving toward this shape but still lacks:

- full native workflow/supervisor/runtime parity
- native/provider-specific HTTP embedding adapter execution and approximate/vector-database persistence beyond the validated adapter config, command transport, reusable provider profiles, and sharded provider-validated weighted vector store

## Success Condition

Clasp is ready to host an autonomous development swarm when all of the following are true:

- `claspc` is fully native and self-hosted
- backend execution is fully native
- the swarm manager/control plane runs natively
- worker roles persist and coordinate through native runtime state
- iterative rebuilds are fast enough for continuous self-improvement
- AppBench-style optimization loops can run without external orchestration glue
